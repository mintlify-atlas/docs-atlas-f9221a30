**gstack turns Claude Code from one generic assistant into a team of specialists you can summon on demand.**

Eight opinionated workflow skills for planning, review, shipping, browser automation, QA testing, and retrospectives — all as slash commands.

## The Problem

Without gstack:
- The agent takes your request literally — it never asks if you're building the right thing
- It will implement exactly what you said, even when the real product is something bigger
- "Review my PR" gives inconsistent depth every time
- "Ship this" turns into a long back-and-forth about what to do
- The agent can write code but can't see your app — it's half blind
- You still do QA by hand: open browser, click around, check pages, squint at layouts

## The Solution

| Skill | Mode | What it does |
|-------|------|------------|
| `/plan-ceo-review` | Founder / CEO | Rethink the problem. Find the 10-star product hiding inside the request. |
| `/plan-eng-review` | Eng manager / tech lead | Lock in architecture, data flow, diagrams, edge cases, and tests. |
| `/review` | Paranoid staff engineer | Find the bugs that pass CI but blow up in production. Triages Greptile review comments. |
| `/ship` | Release engineer | Sync main, run tests, resolve Greptile reviews, push, open PR. For a ready branch, not for deciding what to build. |
| `/browse` | QA engineer | Give the agent eyes. It logs in, clicks through your app, takes screenshots, catches breakage. Full QA pass in 60 seconds. |
| `/qa` | QA lead | Systematic QA testing. On a feature branch, auto-analyzes your diff, identifies affected pages, and tests them. Also: full exploration, quick smoke test, regression mode. |
| `/setup-browser-cookies` | Session manager | Import cookies from your real browser (Comet, Chrome, Arc, Brave, Edge) into the headless session. Test authenticated pages without logging in manually. |
| `/retro` | Engineering manager | Team-aware retro: your deep-dive + per-person praise and growth opportunities for every contributor. |

## When to Use Each Skill

### Planning Phase

Start with **plan mode** before writing code:

1. **`/plan-ceo-review`** - Use when you want to validate product direction
   - You're starting a new feature and want to think big
   - The request seems simple but might be hiding a better product
   - You want founder-mode thinking: "What's the 10-star version?"

2. **`/plan-eng-review`** - Use after product direction is locked
   - You need to nail architecture, data flow, and edge cases
   - You want diagrams that force hidden assumptions into the open
   - You're ready to make the idea buildable, not just exciting

### Implementation Phase

While building:

3. **`/review`** - Use before shipping
   - Catch bugs that pass CI but blow up in production
   - Find N+1 queries, race conditions, trust boundaries
   - Triage Greptile review comments automatically

4. **`/ship`** - Use when the branch is ready
   - Sync main, run tests, bump version, update changelog
   - Handle Greptile comments, push, create PR
   - One command from "code done" to "PR open"

### Testing Phase

After implementation:

5. **`/browse`** - Use for ad-hoc browser testing
   - Log into your app and click through flows
   - Take annotated screenshots for bug reports
   - Check console errors, verify deployments

6. **`/qa`** - Use for systematic testing
   - **Diff-aware mode** (automatic on feature branches): analyzes git diff, identifies affected pages, tests them
   - **Full mode**: systematic exploration of the entire app
   - **Quick mode**: 30-second smoke test
   - **Regression mode**: compare against baseline

7. **`/setup-browser-cookies`** - Use before testing authenticated pages
   - Import real browser sessions into headless browser
   - Skip manual login flows in QA

### Reflection Phase

End of week:

8. **`/retro`** - Use for weekly engineering retrospectives
   - Analyzes commit history, work patterns, shipping velocity
   - Per-person breakdown with specific praise and growth areas
   - Tracks metrics over time, detects trends

## How Skills Work Together

Here's a complete workflow using multiple skills:

```
You:   [enter plan mode]
       I want to add seller photo upload to the listing app.

You:   /plan-ceo-review

Claude: "Photo upload" is not the feature. The real job is helping
        sellers create listings that actually sell. Here's the 10-star
        version: auto-identify product from photo, pull specs and pricing
        comps from web, draft title and description...

You:   /plan-eng-review

Claude: [Architecture diagram, state machine for upload → classify →
        enrich → draft pipeline, async job boundaries, failure modes,
        test matrix]

You:   [exit plan mode, implement the plan]

You:   /review

Claude: Race condition: two tabs can overwrite cover-photo selection.
        Trust boundary: pulling web data into draft generation creates
        a prompt injection vector. Missing: orphan cleanup for failed
        uploads.

You:   (fix the issues)

You:   /ship

Claude: [Syncs main, runs tests, pushes branch, opens PR — 6 tool
        calls, done]

You:   /qa

Claude: Analyzing branch diff... 8 files changed, 3 routes affected.
        [Tests /listings/new, /listings/:id, /api/listings]
        All 3 routes working. Upload + enrichment flow passes end to end.

You:   /setup-browser-cookies staging.myapp.com

Claude: Imported 8 cookies for staging.myapp.com from Chrome.

You:   /qa https://staging.myapp.com --quick

Claude: [Smoke test: homepage + 5 pages, 30 seconds]
        Health Score: 91/100. No critical issues. 1 medium: mobile
        nav overlap.
```

## Explicit Gears

The core philosophy: **planning is not review. Review is not shipping. Founder taste is not engineering rigor.**

If you blur all of that together, you usually get a mediocre blend of all four.

gstack gives you explicit gears. You tell the model what kind of brain you want right now:

- **Founder mode** (`/plan-ceo-review`) - Dream big, find the 10-star product
- **Eng manager mode** (`/plan-eng-review`) - Make it buildable with diagrams and edge cases
- **Paranoid reviewer mode** (`/review`) - Find what can still break
- **Release machine mode** (`/ship`) - Execute the final mile
- **QA engineer mode** (`/browse`, `/qa`) - Give the agent eyes to test
- **Engineering manager mode** (`/retro`) - Analyze what happened and level up

Each skill is optimized for one specific cognitive mode. That's the unlock.
