**This is your engineering manager mode.** At the end of the week you want to know what actually happened. Not vibes — data.

`/retro` analyzes commit history, work patterns, and shipping velocity and writes a candid retrospective.

## Team-Aware Analysis

It is **team-aware**. It identifies who is running the command, gives you the deepest treatment on your own work, then breaks down every contributor with specific praise and growth opportunities — the kind of feedback you would actually give in a 1:1.

### Your Week (Personal Deep-Dive)

For the current user (from `git config user.name`), this section gets the deepest treatment:

- Personal commit count, LOC, test ratio
- Session patterns and peak hours
- Focus areas
- Biggest ship
- **What you did well** (2-3 specific things anchored in commits)
- **Where to level up** (1-2 specific, actionable suggestions)

### Team Breakdown

For each teammate (sorted by commits descending), write:

- **What they shipped**: 2-3 sentences on contributions, areas of focus, commit patterns
- **Praise**: 1-2 specific things they did well, anchored in actual commits. Examples:
  - "Cleaned up the entire auth module in 3 small, reviewable PRs — textbook decomposition"
  - "Added integration tests for every new endpoint, not just happy paths"
  - "Fixed the N+1 query that was causing 2s load times on the dashboard"
- **Opportunity for growth**: 1 specific, constructive suggestion. Frame as investment, not criticism. Examples:
  - "Test coverage on the payment module is at 8% — worth investing in before the next feature lands on top of it"
  - "3 of the 5 PRs were 800+ LOC — breaking these up would catch issues earlier and make review easier"
  - "All commits land between 1-4am — sustainable pace matters for code quality long-term"

**Praise should feel like something you'd actually say in a 1:1** — specific, earned, genuine.

**Growth suggestions should feel like investment advice** — "this is worth your time because..." not "you failed at..."

## Metrics Tracked

| Metric | Description |
|--------|-------------|
| Commits to main | Total commits in the window |
| Contributors | Number of unique authors |
| PRs merged | Extracted from commit messages (#NNN) |
| Total insertions | Lines added |
| Total deletions | Lines removed |
| Net LOC added | Insertions - deletions |
| Test LOC (insertions) | Test files only |
| Test LOC ratio | Test insertions / total insertions |
| Version range | From VERSION file, first to last |
| Active days | Days with at least 1 commit |
| Detected sessions | Coding sessions (45-min gap threshold) |
| Avg LOC/session-hour | Net LOC / active coding time |
| Greptile signal | (fixes + already-fixed) / (fixes + already-fixed + FPs) |

Sort by commits descending. The current user always appears first, labeled "You (name)".

## Greptile Integration

Reads `~/.gstack/greptile-history.md` (if it exists) and filters entries within the retro time window.

Counts entries by type: `fix`, `fp`, `already-fixed`.

Computes signal ratio: `(fix + already-fixed) / (fix + already-fixed + fp)`

If no entries exist in the window or the file doesn't exist, skips the Greptile metric row.

## Commit Time Distribution

Shows hourly histogram in Pacific time using bar chart:

```
Hour  Commits  ████████████████
 00:    4      ████
 07:    5      █████
 ...
```

Identifies and calls out:
- Peak hours
- Dead zones
- Whether pattern is bimodal (morning/evening) or continuous
- Late-night coding clusters (after 10pm)

## Work Session Detection

Detects sessions using **45-minute gap** threshold between consecutive commits.

For each session reports:
- Start/end time (Pacific)
- Number of commits
- Duration in minutes

Classifies sessions:
- **Deep sessions** (50+ min)
- **Medium sessions** (20-50 min)
- **Micro sessions** (&lt;20 min, typically single-commit fire-and-forget)

Calculates:
- Total active coding time (sum of session durations)
- Average session length
- LOC per hour of active time

## Streak Tracking

Counts consecutive days with at least 1 commit to `origin/main`, going back from today.

Tracks both team streak and personal streak:

```bash
# Team streak: all unique commit dates (Pacific time)
TZ=America/Los_Angeles git log origin/main --format="%ad" --date=format:"%Y-%m-%d" | sort -u

# Personal streak: only the current user's commits
TZ=America/Los_Angeles git log origin/main --author="<user_name>" --format="%ad" --date=format:"%Y-%m-%d" | sort -u
```

Count backward from today — how many consecutive days have at least one commit?

Display both:
- "Team shipping streak: 47 consecutive days"
- "Your shipping streak: 32 consecutive days"

## Persistent History & Trend Tracking

Before saving the new snapshot, checks for prior retro history:

```bash
ls -t .context/retros/*.json 2>/dev/null
```

**If prior retros exist:** Load the most recent one. Calculate deltas for key metrics and include a **Trends vs Last Retro** section:

```
                    Last        Now         Delta
Test ratio:         22%    →    41%         ↑19pp
Sessions:           10     →    14          ↑4
LOC/hour:           200    →    350         ↑75%
Fix ratio:          54%    →    30%         ↓24pp (improving)
Commits:            32     →    47          ↑47%
Deep sessions:      3      →    5           ↑2
```

**If no prior retros exist:** Skip the comparison section and append: "First retro recorded — run again next week to see trends."

## Arguments

- `/retro` — default: last 7 days
- `/retro 24h` — last 24 hours
- `/retro 14d` — last 14 days
- `/retro 30d` — last 30 days
- `/retro compare` — compare current window vs prior same-length window
- `/retro compare 14d` — compare with explicit window

## Compare Mode

When you run `/retro compare` (or `/retro compare 14d`):

1. Compute metrics for the current window (default 7d)
2. Compute metrics for the immediately prior same-length window (e.g., 7-14 days ago)
3. Show a side-by-side comparison table with deltas and arrows
4. Write a brief narrative highlighting the biggest improvements and regressions
5. Save only the current-window snapshot to `.context/retros/`

## Top 3 Team Wins

Identify the 3 highest-impact things shipped in the window across the whole team. For each:
- What it was
- Who shipped it
- Why it matters (product/architecture impact)

## 3 Things to Improve

Specific, actionable, anchored in actual commits. Mix personal and team-level suggestions. Phrase as "to get even better, the team could..."

## 3 Habits for Next Week

Small, practical, realistic. Each must be something that takes &lt;5 minutes to adopt. At least one should be team-oriented (e.g., "review each other's PRs same-day").

## Tone

- Encouraging but candid, no coddling
- Specific and concrete — always anchor in actual commits/code
- Skip generic praise ("great job!") — say exactly what was good and why
- Frame improvements as leveling up, not criticism
- **Praise should feel like something you'd actually say in a 1:1** — specific, earned, genuine
- **Growth suggestions should feel like investment advice** — "this is worth your time because..." not "you failed at..."
- Never compare teammates against each other negatively. Each person's section stands on its own.

## When to Use It

Use `/retro` when:

- It's the end of the week and you want to see what actually happened
- You want to track shipping velocity and code quality over time
- You want per-person feedback for the team
- You want to compare this week vs last week

**Do not use it when:**

- You're in the middle of a feature (wait until end of week)
- You want real-time feedback on current work (use `/review` instead)
