## Overview

gstack is powerful with one Claude Code session. It is **transformative with ten**.

[Conductor](https://conductor.build) runs multiple Claude Code sessions in parallel — each in its own isolated workspace. That means you can have one session running `/qa` on staging, another doing `/review` on a PR, a third implementing a feature, and seven more working on other branches. All at the same time.

## What is Conductor?

Conductor is a workspace orchestrator for Claude Code. It manages multiple parallel Claude Code sessions, each in its own git worktree with complete isolation. Think of it as "tmux for AI agents."

## Why Use It with gstack?

gstack gives you specialized cognitive modes — founder taste, engineering rigor, paranoid review, QA testing. Conductor lets you run all of them simultaneously:

- Session 1: `/plan-ceo-review` for a new feature
- Session 2: `/review` on an open PR
- Session 3: `/ship` to land a finished branch
- Session 4: `/qa` testing staging
- Session 5-10: Building different features in parallel

**One person, ten parallel agents, each with the right cognitive mode for its task.** That is not incremental improvement. That is a different way of building software.

## How gstack Works with Multiple Workspaces

Each Conductor workspace is a separate git worktree. gstack automatically detects this and ensures complete isolation:

### Isolated Browser Instances

Each workspace gets its own isolated browser instance automatically:

- **Separate Chromium process** — No shared state between workspaces
- **Independent cookies** — Different sessions, different auth states
- **Per-workspace tabs** — Browser state lives in `.gstack/` inside each project root
- **Dedicated logs** — Debug output isolated per workspace
- **No port collisions** — Browser daemon binds to workspace-specific ports
- **No configuration needed** — Works automatically, zero setup

`/browse` and `/qa` sessions **never interfere with each other**, even across ten parallel workspaces. One workspace can be testing staging.myapp.com while another tests localhost:3000, with completely separate cookie jars and session state.

### State Isolation

All gstack state lives in `.gstack/` inside each workspace:

```
project/
├── .gstack/
│   ├── browser-session/        # Chromium user data dir
│   ├── qa-reports/             # QA test results
│   ├── greptile-history.md     # False positive tracking
│   └── cookies/                # Imported browser cookies
```

Every workspace maintains its own:
- Browser cookies and localStorage
- QA report history
- Greptile false positive database
- Cookie import state

## Conductor Lifecycle Hooks

Conductor automatically manages gstack setup through `conductor.json`:

```json
{
  "setup": "bin/dev-setup",
  "archive": "bin/dev-teardown"
}
```

When Conductor creates a new workspace:
1. Copies `.env` from main worktree (API keys carry over)
2. Runs `bun install`
3. Symlinks skills for dev mode
4. Creates isolated `.gstack/` directory

When archiving a workspace:
1. Removes skill symlinks
2. Cleans up `.claude/` directory
3. Preserves `.gstack/` for history

## No Configuration Needed

**Just works.** No port configuration, no browser profiles to manage, no cookie conflicts to debug. Each workspace is completely independent from the moment Conductor creates it.
