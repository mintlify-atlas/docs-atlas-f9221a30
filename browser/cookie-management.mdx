## Overview

The browser can import cookies from your real browsers (Comet, Chrome, Arc, Brave, Edge) to test authenticated pages without logging in manually through the headless browser.

## Supported browsers

- **Comet** (Chromium-based)
- **Chrome** (Google Chrome)
- **Arc** (Arc Browser)
- **Brave**
- **Edge** (Microsoft Edge)

<Note>
Only macOS is currently supported. Linux and Windows cookie decryption are architecturally possible but not yet implemented.
</Note>

## How it works

<Steps>
  <Step title="Locate cookie database">
    gstack finds your browser's SQLite cookie database in `~/Library/Application Support/`
  </Step>
  <Step title="Copy to temp file">
    Copies the database to `/tmp` to avoid SQLite lock conflicts with the running browser
  </Step>
  <Step title="Read encrypted cookies">
    Opens the database read-only and queries encrypted cookie values
  </Step>
  <Step title="Decrypt via Keychain">
    On first import per browser, prompts for Keychain access to get the encryption key (PBKDF2 + AES-128-CBC)
  </Step>
  <Step title="Load into Playwright">
    Decrypts cookies in memory and loads them into the Playwright browser context
  </Step>
</Steps>

## Interactive picker

The safest way to import cookies:

```bash
$B cookie-import-browser
```

This opens a web UI at `http://localhost:<port>/cookie-picker` where you can:

1. Select your browser (auto-detected)
2. Search for domains
3. See cookie counts per domain (cookie values are never displayed)
4. Import specific domains
5. Remove previously imported domains

<Tip>
The picker UI never displays cookie values — only domain names and counts. This is by design for security.
</Tip>

## Direct import

Skip the UI and import a specific domain directly:

```bash
# Import cookies for github.com from Chrome
$B cookie-import-browser chrome --domain github.com

# Import from Arc
$B cookie-import-browser arc --domain staging.myapp.com

# Auto-detect browser
$B cookie-import-browser --domain example.com
```

## First import per browser

The first time you import cookies from a browser, macOS will prompt for Keychain access:

```
"browse" wants to use your confidential information stored in "Chrome Safe Storage" in your keychain.

To allow this, enter the "login" keychain password.
```

Click **"Always Allow"** to avoid repeated prompts, or **"Allow"** for one-time access.

<Note>
The encryption key is cached in memory for the server's lifetime. When the server shuts down (idle timeout or explicit stop), the cache is cleared.
</Note>

## Security model

### Keychain access requires user approval

First cookie import per browser triggers a macOS Keychain dialog. The user must click "Allow" or "Always Allow." gstack never silently accesses credentials.

### Decryption happens in-process

Cookie values are decrypted in memory (PBKDF2 + AES-128-CBC), loaded into the Playwright context, and never written to disk in plaintext.

### Database is read-only

gstack copies the Chromium cookie DB to a temp file (to avoid SQLite lock conflicts with the running browser) and opens it read-only. It never modifies your real browser's cookie database.

### No cookie values in logs

Console, network, and dialog logs never contain cookie values. The `cookies` command outputs cookie metadata (domain, name, expiry) but values are truncated.

### Key caching is per-session

The Keychain password + derived AES key are cached in memory for the server's lifetime. When the server shuts down (idle timeout or explicit stop), the cache is gone.

## Common workflows

### Testing authenticated pages

```bash
# 1. Import cookies for your staging environment
$B cookie-import-browser chrome --domain staging.myapp.com

# 2. Navigate to authenticated page
$B goto https://staging.myapp.com/dashboard

# 3. Verify you're logged in
$B snapshot -i
# Should show dashboard content, not login form

# 4. Take screenshot
$B screenshot /tmp/dashboard.png
```

### Using with /qa skill

```bash
# Import cookies first
$B cookie-import-browser chrome --domain app.example.com

# Then run QA (cookies persist for the session)
/qa https://app.example.com
```

### Multiple domains

```bash
# Import cookies for multiple domains
$B cookie-import-browser chrome --domain app.example.com
$B cookie-import-browser chrome --domain api.example.com
$B cookie-import-browser chrome --domain cdn.example.com

# Or use interactive picker to select all at once
$B cookie-import-browser
```

## Troubleshooting

### "Keychain access denied"

You clicked "Deny" on the Keychain prompt. Run the command again and click "Allow."

### "No cookies found for domain"

Make sure you're logged in to that domain in your real browser. Check the domain name (should be just `example.com`, not `https://example.com`).

### "Database is locked"

Your browser is running and has the cookie database locked. gstack should handle this automatically by copying to `/tmp` first. If this error persists, close the browser and try again.

### Cookies not working after import

Some sites use additional authentication beyond cookies (CSRF tokens, localStorage, etc.). Cookie import handles HTTP-only and secure cookies but can't transfer localStorage or sessionStorage.

## Import from JSON file

If you have cookies in JSON format (e.g., from browser DevTools):

```bash
$B cookie-import /path/to/cookies.json
```

**Expected format:**
```json
[
  {
    "name": "session",
    "value": "abc123",
    "domain": ".example.com",
    "path": "/",
    "expires": 1735689600,
    "httpOnly": true,
    "secure": true
  }
]
```

The `domain` field is auto-filled from the current page URL if omitted.
